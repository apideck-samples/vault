# Research: vault — OAuth CSRF Fix (GH-9546)

## Research Question

How should the vault repo's OAuth callback page and connection detail page be updated to support the two-phase OAuth confirmation flow? The callback page (`/oauth/callback`) must relay a `confirm_token` to the popup opener via postMessage, and the connection detail page must extract `confirm_token` from URL params and call the confirm endpoint for the redirect-based flow.

## Summary

The vault repo serves two roles in the OAuth flow:
1. **Callback page** (`/oauth/callback`): Used by the popup-based flow (vault-core, platform-admin-v2). After OAuth completes, Unify redirects the popup to this page. Currently it just calls `window.close()`. It needs to be updated to read `confirm_token` from URL params, postMessage it to `window.opener`, then close.
2. **Connection detail page** (`/integrations/[unified-api]/[provider]`): Used by the redirect-based flow. Vault redirects the entire page through the OAuth flow, and Unify redirects back to this page. Currently the page just re-fetches the connection via SWR. It needs to extract `confirm_token` from URL params, call the confirm endpoint with session auth, then refresh.

## Cross-Repo Context

- **Execution order**: Second — depends on Unify (confirm_token in redirect URL, confirm endpoint)
- **Depends on**: unify (provides `confirm_token` in callback redirect URL, provides confirm endpoint)
- **Depended on by**: vault-core (popup lands on vault's `/oauth/callback` which must postMessage the confirm_token), platform-admin-v2 (same popup callback page)
- **API contracts**:

  **Inbound from Unify callback redirect**:
  - URL: `https://vault.apideck.com/oauth/callback?confirm_token=<uuid>` (popup flow)
  - URL: `https://vault.apideck.com/integrations/{api}/{provider}?confirm_token=<uuid>` (redirect flow)

  **Outbound postMessage from callback page** (NEW):
  ```
  window.opener.postMessage({ type: 'oauth-confirm', confirmToken: '<uuid>' }, '*')
  ```
  Received by: vault-core (inside iframe-vault) or platform-admin-v2

  **Outbound confirm API call from connection page** (NEW):
  ```
  POST https://unify.apideck.com/vault/oauth/confirm
  Headers: Authorization: Bearer <jwt>, X-APIDECK-APP-ID, X-APIDECK-CONSUMER-ID, X-APIDECK-AUTH-TYPE: JWT
  Body: { "confirm_token": "<uuid>" }
  ```

## Implementation Details

### 1. Callback Page Modification (`/oauth/callback`)

**Current behavior** (`src/pages/oauth/callback.tsx`):
- `getServerSideProps` checks if query params exist → sets `hasError`
- If no error: `useEffect` calls `window.close()` on mount (line 14)
- If error: shows `OAuthErrorAlert` component

**Required changes**:
- `getServerSideProps`: extract `confirm_token` from query params, pass as prop (separate from error params)
- Client component: if `confirm_token` exists AND `window.opener` exists:
  1. `window.opener.postMessage({ type: 'oauth-confirm', confirmToken }, '*')`
  2. Then `window.close()`
- If `confirm_token` exists but NO `window.opener` (should not happen for popup flow, but defensive):
  1. Just `window.close()`
- If no `confirm_token` (backward compatibility during transition):
  1. Current behavior: `window.close()`

**Important**: The `confirm_token` query param must NOT be treated as an error. Current logic treats ANY query params as an error. This must be updated to distinguish `confirm_token` from error params (`error`, `error_description`).

### 2. Connection Detail Page Modification (redirect flow)

**Current behavior** (`src/pages/integrations/[unified-api]/[provider].tsx`):
- Fetches connection via SWR on mount
- Displays ConnectionForm component
- After OAuth redirect, page reloads, SWR re-fetches connection

**Required changes in `[provider].tsx`**:
- Read `confirm_token` from `router.query` on mount
- Pass it to ConnectionForm as a prop

**Required changes in `ConnectionForm.tsx`**:
- Accept `confirmToken` prop
- On mount, if `confirmToken` is present:
  1. Call `POST /vault/oauth/confirm` via Axios with session auth headers + confirm_token body
  2. On success: trigger SWR revalidation, clear `confirm_token` from URL (using `router.replace`)
  3. On error: show toast notification via existing error handling

**Auth headers for confirm call** (same as existing API calls):
- `Authorization: Bearer ${session.jwt}`
- `X-APIDECK-APP-ID: ${session.applicationId}`
- `X-APIDECK-CONSUMER-ID: ${session.consumerId}`
- Note: `X-APIDECK-AUTH-TYPE: JWT` is already set as a default on the Axios client instance (`lib/axios.ts:6`) — do NOT add it per-request to avoid confusion

### 3. URL Query Param Handling

The callback redirect URL from Unify will be:
- Popup flow: `https://vault.apideck.com/oauth/callback?confirm_token=<uuid>`
- Redirect flow: `https://vault.apideck.com/integrations/{api}/{provider}?confirm_token=<uuid>`

For the redirect flow, the `redirect_uri` is constructed in `ConnectionForm.tsx:197-220`:
```typescript
let redirectUrl = `${window.location.origin}/integrations/${unifiedApi}/${serviceId}`
```
Unify will append `?confirm_token=<uuid>` to this. No changes needed to URL construction — the connection page just needs to read `confirm_token` from the resulting query params.

### 4. Error Handling

**Callback page**: If postMessage fails (e.g., opener already closed), silently close the window. The pending tokens will expire via TTL.

**Connection page**: If confirm API call fails:
- 403 (consumer mismatch): Show error toast — "OAuth authorization could not be confirmed. Please try again."
- 404 (expired/not found): Show error toast — "OAuth confirmation expired. Please authorize again."
- Network error: Show generic error toast

### 5. Backward Compatibility

With the detect-and-remediate model, backward compatibility is inherent:
- Callback page: if no `confirm_token` in URL, behave as before (just close). Tokens are already stored at callback time.
- Connection page: if no `confirm_token` in URL, behave as before (just SWR refresh). Tokens are already stored at callback time.
- When `confirm_token` IS present and confirm succeeds: tokens were already stored (no visible change). Pending record cleaned up.
- When confirm fails (403 mismatch): tokens are revoked server-side. Show error toast.

### 6. Test Infrastructure

**Important**: `__tests__/unit/pages/oauth/` directory does NOT exist — it must be created for callback page tests. Additionally, `window.opener` is not available in jsdom by default and needs explicit mocking:
```typescript
Object.defineProperty(window, 'opener', { value: mockOpener, writable: true })
```

### 7. getServerSideProps Redirect Collision

**Verify**: `[provider].tsx` lines 137-158 contain `redirectToAppUrl` logic that checks query params. Ensure this logic does not interfere when `confirm_token` is present in the URL. The `redirectToAppUrl` check should not treat `confirm_token` as a redirect trigger.

## Scope & Constraints

**In scope:**
- Modify `/oauth/callback` page to read `confirm_token` and postMessage to opener
- Modify connection detail page to extract `confirm_token` from URL and call confirm endpoint
- Backward compatibility when `confirm_token` is absent
- Error handling for confirm failures

**Out of scope:**
- Changing the OAuth initiation flow (redirect_uri construction stays the same)
- Origin validation on postMessage (pre-existing issue, separate concern)
- Popup-based flow in vault (vault uses redirect-based only; popup flow is vault-core's responsibility)
